# Product Notes

## Problem

AI workflow tools need to show more than model output. A useful operator surface also needs source inputs, connector checks, confidence thresholds, review paths, and product feedback from repeated usage.

## Product Shape

The main page is structured as a study, not a dashboard:

1. Start with the research question: when is an AI workflow ready for a customer pilot?
2. State the hypothesis framework: workflow reality, contract checks, evaluation, and product signal.
3. Show what was built in code: typed synthetic workflows, readiness scoring, blocker detection, repair, and recommendation generation.
4. Let the reader run the experiment through `Run checks`, `Repair issue`, and `Generate brief`.
5. End with recommendations and learnings.

The interface is still interactive, but the interaction is evidence inside the study. The reader should understand the investigation before they touch the controls.

## Data Boundary

The customer scenarios, workflows, metrics, and source records are simulated for the case study. The project is designed to run without secrets or external API keys.
